- **Task:** Should we adopt a background job queue, and if so which?

## Recommendation
Adopt **BullMQ on the existing Redis**. Move bulk exports, transactional email,
and webhook fan-out off the request path. Ship behind a feature flag; cut over
one job type at a time starting with exports (highest pain, lowest risk).

## Reasoning trail
- Request handlers were timing out on bulk exports → clear need for async work.
- Redis is already operated for sessions → no new infrastructure to learn.
- BullMQ gives retries, scheduling, and a dead-letter queue out of the box.
- Alternatives considered: a managed queue (SQS) — rejected for now as new infra
  and egress cost for a problem the existing Redis solves; a cron-scraped DB
  table — rejected as a queue we'd have to reinvent badly.

## Dissents (preserved)
- **security-engineer:** Treating Redis as a queue makes it a *correctness*
  dependency, not a cache. Without persistence + auth on the Redis instance, a
  flush silently drops in-flight jobs. Gate launch on this. *(Not blocking the
  decision; blocking the launch.)*
- **qa-engineer:** No merge until each job type has a test proving idempotency
  under double-delivery. Retries are a feature only if jobs are idempotent.
