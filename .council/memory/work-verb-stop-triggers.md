# Memory: Work Verb Stop Triggers

Durable semantics for how a `work` session stops. The principle: a named trigger must be a fireable trigger — no phantoms, no false guardrails.

## Decision
A `work` session has **five** stop triggers — chair-done, `max_turns`, `scratch_max_bytes`, `max_wall_seconds`, and user-stop — and it ends when the first fires. Three are hard and deterministically enforceable: **budget** = `max_turns` (an exact turn count), **scratch-size** = `scratch_max_bytes` (exact, via `wc -c`), and **wall-clock** = `max_wall_seconds`. The other two are external signals (chair-done, user-stop). **`max_wall_seconds` is optional and armed only when present and `> 0`** (`absent | 0 | negative → unarmed`; absence is the off-switch — most councils don't set it, and there is no inert/commented line in any config). When armed it fires at a **turn boundary** as `now − Started(epoch) ≥ max_wall_seconds`, where `now` is a fresh `date +%s` and `Started(epoch)` is a `date +%s` recorded into the scratchpad header at session open. The honest caveat: it is exact only at turn-boundary granularity — a turn already running finishes first, so total wall time can overshoot the budget by up to one turn's duration (it bounds when the *next* turn starts, not a mid-turn cut). Tie-break among simultaneous triggers follows the step-6c bullet order. **`max_tokens` was removed entirely** (not merely softened): the orchestrator has no reliable per-turn token count, so a token budget — hard *or* soft — can never fire, and worse, it was **displayed in the run banner** as `Budget: max_tokens 250k`, i.e. a spend cap the user could see but the run could never honor. By this topic's own principle (*a named trigger must be a fireable trigger*) that is a false guardrail, and a *displayed* one is worse than a merely-named one. So `max_tokens` is gone from all 8 config files, the banner, SKILL.md, and PLAN.md; token spend now rides along with `max_turns` (the run-length cap), documented as a cost expectation, not a trigger. **Wall-clock was cut in Phase 3 as a phantom** (named but with no backing config field, so it could never fire) — then **re-added in Phase 4 as a real fireable trigger**, this time *with* a backing field (`max_wall_seconds`) and the `date +%s` epoch mechanism above, so it is now genuinely measurable, not a phantom. The history is cut-then-properly-restored, not a reversal: Phase 3 was right to cut an unbacked name, and Phase 4 is right to ship the name only once it has a mechanism that honors this topic's principle (*a named trigger must be a fireable trigger*). The `work_budget` comment lines are pinned identically across all six templates and both live `council.yaml`s; the `work` scratchpad header is pinned verbatim (`# Scratchpad — work`, chair-selected Seats line, `## Turn N — <seat>` convention, no user-input section) and anchored by a `Mode: work` worked example.

**Resolution (done).** The review recommended demoting `max_tokens`; the user went further and **removed it outright** in favour of `scratch_max_bytes` + `max_turns` — cleaner, since there's then no phantom knob to mis-display at all. Landed on `main` alongside the Phase-3 merge. **Metric:** across real `work` runs the recorded stop reason is always one of {chair-done, max_turns, scratch_max_bytes, max_wall_seconds, user-stop}, and no budget field names a number that can't halt the run (`max_wall_seconds` counts here only when armed — when set `> 0`; an unset/0/negative value is not a phantom because it is never displayed as an active cap).

## Why
A guardrail you can't configure or observe isn't a guardrail. Wall-clock had no config field, so naming it only misled; `max_turns` already bounds the dimension we actually care about — total iterations / work, counted exactly — so a separate wall-clock knob would add an enforcement semantic we'd have to invent for little gain. (Note the honest limit: `max_turns` is an *iteration/cost* cap, not a tight *wall-time* bound — a single turn can run arbitrarily long — so it's a loose time proxy, which is why a real `max_wall_seconds` could return in Phase 4 *if* a user showed turns were a poor time proxy. **That door is now closed: it returned in Phase 4** — the deciding signal was the user's own explicit `PLAN.md` edit re-adding wall-clock as a real, fireable trigger, which is exactly the "user shows they want it" evidence the door was held open for. It ships **absent-by-default** (no inert config line — absence is the off-switch), so it adds a real capability without adding a phantom or a surface every council must carry.) `max_tokens` is worse than absent for the same reason squared: resting any stop on a rough, unobservable estimate fires early (wastes paid budget), late (overshoots), or never — and displaying it as "Budget" tells the user a cap exists when it can't. The honest fix in every case was to make the spec tell the truth — cut/remove the unfireable knob — not to build mechanism (e.g. mid-run token accounting) to make a word true that the user never asked for. The header/example/comment fixes exist because un-pinned formats and disagreeing fixtures drift and erode after-the-fact auditability. See [[council-record-and-memory-format]] for the format these artifacts conform to.

## Standing dissent (qa-engineer)
No live dissent — QA and PM converged on cut-wall-clock and (after the review) removing `max_tokens`. The durable tension to carry into the next change to these triggers: QA's drift / un-auditability axis (an unfireable or unpinned trigger is exactly the silent-rot failure to guard against) vs. PM's surface-area / process-overhead axis (don't ship knobs or enforcement you can't cleanly spec). Here both pointed the same way. They will diverge again when a stricter check trades off against a smaller surface — balance them then, and prefer making the spec honest over building mechanism to justify a name.

**Phase 4 — they diverged (as predicted).** PM argued (on the surface-area axis) to *decline* `max_wall_seconds` — the evidence bar from the door wasn't met, and a new budget knob is surface every council carries. QA's drift axis was satisfiable either way (a *backed, fireable* trigger isn't a phantom). PM lost on the **deciding fact**: the user's explicit `PLAN.md` edit re-adding wall-clock *is* the evidence the door was held open for. The reconciliation that honors both axes: the field ships **armed-by-default-off** (absent = unarmed; no inert/commented config line in any of the 8 files), so the capability exists for the user who asked, while the surface stays zero for every council that doesn't set it — PM's surface-area discipline is preserved in the absent-by-default shape, not overridden.
